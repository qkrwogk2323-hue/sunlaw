## 1. 변제예정액 산출 공식 검증

### VS 공식 (`repayment-calculator.ts`)

```
monthlyAvailable = netSalary - (livingCost + extraLivingCost + childSupport)
monthlyAvailable *= (1 - trusteeCommRate / 100)   // 외부위원 수수료 차감
monthlyRepay = min(monthlyAvailable, ceil(targetAmount / repayMonths))
```

- `targetAmount` = `unsecuredDebt` (무담보 채무 총액)
- `repayMonths` = 변제기간 (기본 36개월)
- 청산가치 보장 원칙: `totalRepay >= liquidationValue` 검증 포함

**결과**: ✅ 공식 정확. colaw 산출 로직과 동일한 구조.

## 3. 소득 데이터 갭 수정

### net_salary = 0 문제 (수정 완료)

마이그레이션에서 colaw `monthaverageincomemoney` → VS `gross_salary`로만 매핑하고, `net_salary`에는 값을 넣지 않음. VS UI는 `net_salary`를 사용하므로 변제예정액이 0으로 계산됨.

**영향 범위**: 39건 (소득 데이터가 있는 전체)

**수정**: `UPDATE rehabilitation_income_settings SET net_salary = gross_salary WHERE net_salary = 0 AND gross_salary > 0` 실행 완료.

## 4. 생계비 적용 방식 차이

### colaw vs VS 동작 차이

| 항목 | colaw | VS |
|------|-------|-----|
| 생계비 저장 | DB에 금액 직접 저장 | DB 저장 없음, UI에서 실시간 계산 |
| 계산 기준 | `기준중위소득 60%` (선택된 연도 기준) | `getLivingCost(year, dependentCount)` 함수 |
| 연도 선택 | 사건별 드롭다운으로 지정 | `incomeSettings.median_income_year` 참조 |
| 가구 수 | 가족 수에서 자동 계산 | `family_members`에서 `is_dependent=true` 카운트 |

**핵심**: VS는 DB에 생계비 금액을 저장하지 않고, 매번 `median-income.ts` 테이블에서 조회함. 따라서 `median-income.ts`의 정확성이 모든 사건의 변제예정액에 직접 영향.

## 6. 미해결 사항

### 6-1. 소득 데이터 미등록 사건 (~51건)

90건 중 약 39건만 `income_settings` 데이터가 있음. 나머지 ~51건은 colaw에서도 소득 데이터가 없었거나, 마이그레이션 대상에서 누락된 것으로 추정.

**권장 조치**: colaw에서 해당 사건들의 수입지출 탭 데이터 존재 여부 확인.

### 6-2. median_income_year 미설정 사건

`income_settings.median_income_year`가 NULL인 사건은 VS가 기본값(현재 연도)으로 fallback. 마이그레이션 데이터에서 colaw 연도 선택 값을 가져와야 함.

### 6-3. 수동 채권자 검토 (6건)

`creditor-mismatch-report.md` 참조. 자동 수정 불가한 6건은 담당자 확인 필요.
