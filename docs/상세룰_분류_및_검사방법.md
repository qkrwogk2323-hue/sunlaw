# 상세룰 분류 및 검사방법

이 문서는 `docs/PROJECT_RULES.md`와 `docs/UX_RULES.md`의 공통 규칙을 제외한 상세 운영 규칙을 역할·조직유형·기능군 기준으로 재분류한 실행 문서다.

목표는 아래 3가지다.

1. 이미 구현된 기능이 어느 소속인지 먼저 분류한다.
2. 소속이 섞인 기능을 버그로 판정한다.
3. 신규 기능 구현 전 반드시 무엇을 먼저 정해야 하는지 체크한다.

## 1. 최상위 분류

- `공통 영역`
- `플랫폼 조직`
- `사용자조직`
- `의뢰인 포털`

`공통 영역`은 디자인, UX/UI, 오류, 토스트, 배너, 로그 포맷 같은 공용 기반만 포함한다. 실제 업무 기능은 반드시 `플랫폼 조직`, `사용자조직`, `의뢰인 포털` 중 하나로 귀속한다.

## 2. 사용자조직 하위 분류

- `법무/법률사무소`
- `신용정보회사`
- `일반 기업`
- `기타 조직`

`기타 조직`은 임시 분류가 아니라 아직 전용 모드로 분리되지 않은 사용자조직 묶음이다. 새 업종 요구가 반복되면 별도 유형으로 승격한다.

## 3. 플랫폼 조직 상세룰

### 3-1. 소속 기능

- 조직 신청 접수, 승인, 반려, 취소
- 조직 생성, 비활성화, 삭제
- 구독 권한 부여, 제한, 회수
- 플랫폼 공통 설정
- 전체 감사로그
- 운영지원, 고객센터
- 플랫폼 전용 알림
- 플랫폼 조직 AI

### 3-2. 금지 사항

- 사용자조직 전용 사건·의뢰인·비용·법무 기능을 플랫폼 기본 메뉴에 섞지 않는다.
- 플랫폼 조직에 없는 메뉴를 기본 진입 경로로 사용하지 않는다.
- 타 조직 운영 기능을 문자열 비교나 임의 분기로 숨기지 않는다.

### 3-3. 검사방법

1. 플랫폼 조직 로그인 시 메뉴 첫 주소가 실제 기본 진입 경로와 같은지 확인한다.
2. 플랫폼 조직에서 `조직 신청 관리`, `조직 관리`, `감사로그`, `구독 조정`이 보이는지 확인한다.
3. 일반 사용자조직 메뉴가 플랫폼 조직에 노출되지 않는지 확인한다.
4. 조직 신청 승인·반려·삭제 후 현재 상태와 과거 로그가 각각 보이는지 확인한다.
5. 구독 권한 변경 후 변경 기록과 감사로그가 함께 남는지 확인한다.
6. 플랫폼 알림이 특정 사용자조직 아래로 잘못 귀속되지 않는지 확인한다.

## 4. 사용자조직 공통 상세룰

### 4-1. 소속 기능

- 사건
- 의뢰인
- 문서
- 일정
- 계약
- 비용
- 허브/협업
- 사용자조직 알림
- 사용자조직 공통 AI
- 조직 내부 설정

### 4-2. 금지 사항

- 플랫폼 조직 권능이 사용자조직 메뉴에 보이면 안 된다.
- 업종 전용 기능을 모든 사용자조직에 일괄 노출하면 안 된다.
- 사용자조직 기본 진입 경로가 메뉴에 없는 주소면 안 된다.

### 4-3. 검사방법

1. 사용자조직 로그인 시 플랫폼 조직 전용 버튼과 화면이 노출되지 않는지 확인한다.
2. 기본 진입 경로가 현재 메뉴 첫 주소와 일치하는지 확인한다.
3. 사건, 의뢰인, 문서, 비용, 계약이 같은 조직 컨텍스트로 묶여 보이는지 확인한다.
4. 중요 화면마다 현재 상태와 과거 기록을 볼 수 있는 경로가 있는지 확인한다.
5. 사용자조직 공통 AI가 플랫폼 조직 데이터나 타 조직 데이터를 끌어오지 않는지 확인한다.

## 5. 법무/법률사무소 상세룰

### 5-1. 소속 기능

- 사건 진행 관리
- 기일/절차 흐름
- 문서 검토/제출
- 보정 요청, 보정 권고, 보정 명령
- 회생/파산
- 법원 제출물 생성
- 의뢰인 사건 진행 공유
- 법무 AI

### 5-2. 금지 사항

- 회생/파산을 별도 전역 현황판으로 중복 구현하지 않는다.
- 법원 제출 흐름을 일반 기업 공통 메뉴처럼 노출하지 않는다.
- 법무 전용 AI를 다른 업종 공통 AI처럼 일괄 적용하지 않는다.

### 5-3. 검사방법

1. 법무 전용 메뉴가 신용정보회사, 일반 기업, 기타 조직에 노출되지 않는지 확인한다.
2. 회생/파산은 사건 타입 기준으로만 나타나는지 확인한다.
3. 보정 문서 업로드 결과가 사건 상세, 허브, 의뢰인 포털로 일관되게 연결되는지 확인한다.
4. 법원 제출물 생성 흐름이 사건 원본 데이터와 연동되는지 확인한다.
5. 보정, 회생/파산, 제출 관련 로그가 사건 흐름 안에서 보이는지 확인한다.

## 6. 신용정보회사 상세룰

### 6-1. 소속 기능

- 회수 활동
- 추심 운영
- 보수 산정
- 정산
- 성과 리포트
- 신용정보회사 AI

### 6-2. 금지 사항

- 법무 전용 보정, 회생/파산, 제출 기능을 신용정보회사 공통 메뉴에 섞지 않는다.
- 추심 전용 수치를 일반 사용자조직 공통 KPI로 강제하지 않는다.

### 6-3. 검사방법

1. 추심 운영, 보수, 정산 메뉴가 법무 조직과 일반 기업에 잘못 노출되지 않는지 확인한다.
2. 회수 실적, 보수 산정, 정산의 원본 기록 링크가 존재하는지 확인한다.
3. 추심 AI가 법무 화면이나 의뢰인 포털에 섞여 노출되지 않는지 확인한다.

## 7. 일반 기업 상세룰

### 7-1. 소속 기능

- 사용자조직 공통 기능 우선
- 업종 특화 기능이 없는 기본 운영 흐름

### 7-2. 금지 사항

- 법무 전용 기능과 신용정보회사 전용 기능을 기본 노출하지 않는다.
- 전용 기능이 필요해지면 일반 기업 공통에 억지로 섞지 않고 별도 분류로 승격한다.

### 7-3. 검사방법

1. 법무/추심 전용 메뉴가 일반 기업에 보이지 않는지 확인한다.
2. 일반 기업이 실제로 쓰는 공통 기능만 노출되는지 확인한다.
3. 로그, 오류, 알림은 공통 규칙을 따르되 업종 전용 설명이 섞이지 않는지 확인한다.

## 8. 기타 조직 상세룰

### 8-1. 소속 기능

- 사용자조직 공통 기능만 우선 사용

### 8-2. 금지 사항

- 법무, 신용정보회사 전용 기능을 기본 노출하지 않는다.
- 아직 정의되지 않은 전용 기능을 `기타 조직`에 몰아넣고 끝내지 않는다.

### 8-3. 검사방법

1. 법무/추심 전용 기능이 보이지 않는지 확인한다.
2. 반복되는 업종 특화 요구가 생기면 `기타 조직 유지`인지 `별도 분리`인지 결정했는지 확인한다.

## 9. 의뢰인 포털 상세룰

### 9-1. 소속 기능

- 사건 진행 공유
- 요청 확인
- 문서 확인
- 비용/청구/납부 확인
- 보정/변제계획 등 의뢰인 공유 정보
- 포털 알림
- 의뢰인 설명형 AI

### 9-2. 금지 사항

- 내부 운영 메뉴를 의뢰인에게 노출하지 않는다.
- 내부 관리자용 로그와 권한 버튼을 의뢰인에게 노출하지 않는다.

### 9-3. 검사방법

1. 의뢰인이 로그인했을 때 내부 조직 메뉴가 보이지 않는지 확인한다.
2. 의뢰인 포털 기본 진입이 포털 메뉴 첫 주소와 일치하는지 확인한다.
3. 사건 허브, 비용, 보정, 변제계획 공유 정보가 실제 사건 데이터와 맞는지 확인한다.
4. 의뢰인이 필요한 자료를 언제까지 내야 하는지 자연어로 이해 가능한지 확인한다.

## 10. AI 상세룰

## 10-1. AI 최상위 분류

- `공통 AI`
- `플랫폼 조직 AI`
- `사용자조직 공통 AI`
- `법무 AI`
- `신용정보회사 AI`
- `의뢰인 포털 AI`

## 10-2. 사용자조직 AI 세부분류

- `홈 AI`
- `작성 AI`
- `사건 AI`
- `문서 AI`
- `의뢰인 소통 AI`
- `비용/계약 AI`
- `허브/협업 AI`

## 10-3. 검사방법

1. AI 기능마다 소속 영역과 소속 조직유형이 정해져 있는지 확인한다.
2. 소속되지 않은 화면에 AI가 붙어 있으면 버그로 본다.
3. AI가 링크를 안내하면 실제 현재 메뉴 구조와 맞는지 확인한다.
4. AI가 본 데이터와 실행할 수 있는 액션 범위가 같은 소속 규칙 안에 있는지 확인한다.
5. AI 오답 신고, 요청 ID, 모델 버전, 질문/답변 기록이 남는지 확인한다.

## 11. 로그 상세룰

### 11-1. 기록이 반드시 보여야 하는 대상

- 조직 신청, 승인, 반려, 취소
- 조직 비활성화, 삭제, 복구
- 구독 권한 변경
- 사건 생성, 수정, 상태 변경
- 문서 등록, 수정, 검토, 승인, 공유
- 의뢰인 초대, 연결 요청, 재승인, 비활성화
- 비용, 청구, 입금, 분납, 약정
- 허브 메시지, 허브 참여, 사건 공유
- 구성원 초대, 권한 변경, 구성원 삭제
- 설정 변경
- 의뢰인과의 대화 및 요청

### 11-2. 검사방법

1. 현재 상태 화면과 과거 기록 화면이 분리되어 있는지 확인한다.
2. 각 메뉴에 최소 하나의 `기록 보기`, `감사로그 보기`, `원본 기록 보기` 진입점이 있는지 확인한다.
3. 버튼 이름만 보고 무슨 기록인지 알 수 없으면 버튼 아래 설명을 추가한다.
4. 의뢰인과의 대화, 문서, 요청은 증거성 기록으로 취급해 우선 노출한다.

## 12. 오류 상세룰

### 12-1. 오류 분류

- `권한 오류`
- `입력 검증 오류`
- `상태 충돌 오류`
- `외부 연동 오류`
- `설명 가능한 업무 오류`
- `설명되지 않은 예외`

### 12-2. 검사방법

1. 오류 화면에 최소 `오류 코드`, `차단 이유`, `원인`, `해결 방법`이 나오는지 확인한다.
2. 설명 가능한 오류인데 `예외가 발생했습니다`로만 끝나면 버그로 본다.
3. 일부 데이터 조회 실패로 전체 페이지가 죽으면 버그로 본다.
4. provider 누락, 상태 누락, 컨텍스트 누락으로 전체 화면이 죽으면 버그로 본다.
5. `UNEXPECTED_EXCEPTION`은 설명되지 않은 예외로 분류해 별도 버그 후보 레지스터에 남긴다.

## 13. 메뉴 및 기본 진입 경로 상세룰

### 13-1. 원칙

- 기본 진입 경로는 현재 사용자에게 보이는 메뉴의 첫 주소를 기준으로 삼는다.
- 메뉴에 없는 주소를 기본 진입 경로로 쓰면 버그다.
- 플랫폼 조직, 사용자조직, 의뢰인 포털은 메뉴 세트를 각각 분리한다.
- 사용자조직 내부에서도 조직유형별 메뉴 차이를 둔다.

### 13-2. 검사방법

1. 메뉴 첫 항목과 실제 첫 진입 경로가 같은지 확인한다.
2. 브랜드 배너, 뒤로가기 기본값, 로그인 후 이동, 프로필 완료 후 이동이 같은 기준을 쓰는지 확인한다.
3. 플랫폼 조직에 없는 `대시보드`가 기본 진입으로 뜨면 버그로 본다.

## 14. 신규 구현 전 분류 확정 룰

새 조직 유형, 새 메뉴, 새 AI, 새 업무 기능을 구현하기 전 반드시 아래를 먼저 확정한다.

- 소속 영역
- 대상 사용자
- 조직 유형
- 메뉴 위치
- 기본 진입 경로
- 보여줄 기능
- 숨길 기능
- 로그 노출 위치
- 오류 처리 방식
- AI 소속
- 테스트 소속
- `기타 조직 유지`인지 `별도 유형 분리`인지

위 항목이 비어 있으면 구현을 시작하지 않는다.

## 15. 이미 구현된 기능 검사 순서

기존 기능은 아래 순서로 전수 조사한다.

1. 기능 이름을 적는다.
2. 기능 소속을 `플랫폼 조직 / 사용자조직 / 의뢰인 포털` 중 하나로 확정한다.
3. 사용자조직이면 `법무 / 신용정보회사 / 일반 기업 / 기타` 중 하나로 확정한다.
4. 현재 메뉴 노출 대상을 확인한다.
5. 기본 진입 경로가 메뉴 첫 주소와 일치하는지 확인한다.
6. 권한 경계가 맞는지 확인한다.
7. 로그 진입점이 있는지 확인한다.
8. 오류 설명이 있는지 확인한다.
9. AI가 붙었다면 AI 소속이 맞는지 확인한다.
10. 테스트 소속이 맞는지 확인한다.

하나라도 어긋나면 `기능 혼입 버그`로 기록한다.

## 16. 기능 혼입 버그 판정 기준

아래 중 하나라도 만족하면 기능 혼입 버그다.

- 플랫폼 전용 기능이 사용자조직에 노출됨
- 사용자조직 전용 기능이 플랫폼 조직에 노출됨
- 법무 전용 기능이 신용정보회사, 일반 기업, 기타 조직에 노출됨
- 신용정보회사 전용 기능이 법무 조직, 일반 기업, 기타 조직에 노출됨
- 의뢰인 포털에 내부 운영 기능이 노출됨
- 메뉴에 없는 주소가 기본 진입 경로로 쓰임
- 로그가 있어야 하는 화면에 로그 진입점이 없음
- 설명 가능한 오류가 설명 없는 예외로 노출됨
- AI가 소속되지 않은 화면이나 조직유형에 노출됨
