# 로그관련 사용자 가이드

이 문서는 Vein Spiral의 로그 체계를 개발자/운영자가 헷갈리지 않도록 정리한 실무용 가이드입니다.

## 1) 로그 저장소 구분

### A. 감사로그(트리거 기반)
- 저장 위치: `audit.change_log`
- 수집 방식: DB 트리거 `audit.capture_row_change()`
- 트리거 조건: 대상 테이블에서 `INSERT / UPDATE / DELETE` 발생 시 자동 적재
- 관리자 화면: `/admin/audit`
- 조회 코드: `src/lib/queries/audit.ts`

핵심 스키마(요약):
- `table_name`, `operation`, `actor_user_id`, `organization_id`, `case_id`
- `changed_fields`, `old_values`, `new_values`, `logged_at`

정의 위치:
- `supabase/migrations/0005_audit.sql`

### B. 앱 이벤트 로그(코드 직접 기록)
- 저장 위치: `public.audit_logs` (앱 코드에서 insert)
- 수집 방식: 서버 액션/로직에서 명시적으로 `from('audit_logs').insert(...)`
- 목적: 사용자 행위 이벤트(도메인 액션)를 앱 레벨로 남김

주요 코드 위치:
- `src/lib/actions/case-actions.ts`
- `src/lib/actions/notification-actions.ts`
- `src/lib/actions/support-actions.ts`
- `src/lib/actions/organization-actions.ts`

주의:
- `/admin/audit` 화면은 기본적으로 `audit.change_log`를 조회합니다.
- `audit_logs`는 별도 테이블/별도 용도로, 자동으로 같은 화면에 합쳐지지 않습니다.

## 2) 실제로 어떤 기록이 /admin/audit 에 보이나?

`/admin/audit`는 아래 필터로 `audit.change_log`를 보여줍니다.
- 일반 작업: 전체
- 삭제 기록: `DELETE`, `SOFT_DELETE`, `ARCHIVE`
- 위반 기록: `VIOLATION*` 접두
- 복구 기록: `RESTORE*` 접두

관련 코드:
- `src/app/(app)/admin/audit/page.tsx`

## 3) 새 테이블 추가 시 로그 누락 방지 체크리스트

신규 테이블을 만들면 아래를 반드시 확인합니다.

1. `audit.capture_row_change()` 트리거 추가
- 예시:
```sql
drop trigger if exists audit_<table> on public.<table>;
create trigger audit_<table>
after insert or update or delete on public.<table>
for each row execute procedure audit.capture_row_change();
```

2. `organization_id` / `case_id` 컬럼 의미 정리
- 감사로그 검색성 확보를 위해 가능하면 표준 컬럼명을 유지

3. 운영에서 볼 화면이 필요한지 확인
- `/admin/audit`로 충분한지
- 별도 도메인 이벤트(`audit_logs`)도 필요한지

4. RLS/권한 확인
- 감사로그 조회는 플랫폼 관리자 권한 기반
- 필요 시 정책 변경 동반

## 4) 빠른 점검 명령어

### 감사 트리거 부착 테이블 확인
```bash
rg -n "audit\.capture_row_change\(" supabase/migrations
```

### 관리자 감사로그 조회 경로 확인
```bash
rg -n "listAuditChangeLog|/admin/audit|schema\('audit'\)" src
```

### 앱 이벤트 로그(audit_logs) 사용처 확인
```bash
rg -n "from\('audit_logs'\)" src/lib/actions
```

## 5) 장애 대응 팁

증상: "기록보관함에 로그가 안 보임"

우선순위 점검:
1. 해당 테이블에 `audit.capture_row_change` 트리거가 있는지
2. 실제 변경이 `INSERT/UPDATE/DELETE`로 발생했는지
3. 플랫폼 관리자 권한으로 `/admin/audit`를 보고 있는지
4. 필터(`table`, `actor`, 탭)가 너무 좁지 않은지
5. 앱 이벤트 로그(`audit_logs`)를 기대한 건 아닌지 (조회 소스 분리)

## 6) 변경 원칙

- 데이터 무결성 추적은 `audit.change_log`를 기준으로 유지
- 사용자 행위 시나리오 추적은 `audit_logs`를 보조 채널로 사용
- 신규 도메인 도입 시, "트리거 로그"와 "앱 이벤트 로그"를 의도적으로 분리 설계
