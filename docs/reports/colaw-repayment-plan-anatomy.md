## 1. 변제계획안 실행 아키텍처

- **편집 입력 페이지**: `https://colaw.co.kr/rescureManage/popupRescureIncomeExpenditure?casebasicsseq=...&diaryyear=...&resurapplicationpersonseq=...`
- **채권자 페이지**: `popupRescureCreditorList` (같은 파라미터)
- **신청인 페이지**: `popupRescureApplication` (같은 파라미터) — 여기에 "인쇄 미리보기", "채권자목록", "변제계획안", "채권자+변제계획안", "전체출력" 5개 출력 옵션 버튼 존재
- **10항 탭**: `popupRescurePlanSection` — 이직신고/재산목록제출/수정허가/수시자료/강제집행 등 법적 문구 boilerplate (숫자 계산과 무관)
- **실제 변제계획안 생성**: Clipsoft Report4 서버 템플릿
  - 팝업 URL 프리픽스: `https://report.colaw.co.kr/ClipReport4/JavaOOFGeneratorChang...`
  - 팝업 이름: `clipsoftReortViewer`
  - 파라미터: `lawyercode=20130527120720950`, `diaryyear`, `resurapplicationpersonseq` 등이 `%5E`(^) 구분자로 전달
  - **즉, 변제계획안의 "렌더링 수식"은 서버의 리포트 템플릿 안에 있음**. 프론트 JS는 입력값 정합성 검증과 자동계산 일부만 담당
- **계산 성격 분류**:
  - 입력 원본: 월소득·가족수·연도·채권자·재산 (수동 입력)
  - 클라이언트 보조 계산: 기준중위소득 60% 표 → 생계비 기본값, 증액률 적용값, 연환산 소득, 라이프니츠 계수 표 조회
  - 서버 최종 렌더: 월별 변제표·청산가치·총변제액·변제율 → PDF

**핵심 수식**:

**⚠️ VS 이식 시 주의**:
- VS는 현재 "기준중위소득 60%"만 하드코드. `lowestlivingmoneyrate` 증액률(100~200%)을 사용자가 케이스별로 지정 가능한 UI가 필요.
- `livingmoneycalcumethod` 1/2 분기에 따라 계산 경로가 달라질 수 있음 (추가 조사 필요).
- `usingfamily_low_money`는 **생계비와 별도로 가용소득에서 추가 공제되는 "기준범위 초과 생계비"**. 김한경 케이스는 비어있지만 다른 케이스에서 등장 가능.

- 김한경의 경우 "원금만 변제" + 36개월 + forcingrepaymentmonthoption 미선택 → **자동으로 36개월 계산 결과**가 도출됨. 이것이 서버 측 로직인지 프론트 보조계산인지 확인 필요.

## 4. 생계비 증액률 사례 검증

김한경 (1인 가족, 2026년 기준):
- 기본 60% = 1,025,695원
- 150% 증액 적용 = 1,538,543원
- 검증: 1,025,695 × 1.5 = 1,538,542.5 ≈ 1,538,543 (원 단위 반올림)

**2026년 1인 기준중위소득 역산**:
- 1,025,695 ÷ 0.6 = 1,709,491 → 월 1,709,491원
- 또는 연 20,513,892원 → 보건복지부 2026 고시와 교차검증 필요

**VS 현재 median-income.ts 확인 포인트**:
- 2026년 1인 값이 1,709,491원 부근인지
- 증액률 150%를 입력받아 계산하는 인터페이스가 있는지
- 없다면 신설 필요

## 5. 변제계획안 계산 흐름 (추정 — 서버 로직 미확인)

```
1. 입력: 월평균 소득, 부양가족, 연도, 추가생계비, 양육비, 보수율, 처분재산
2. 기준중위소득 60% 조회 → lowMoney
3. 증액률 적용 → resultlowestlivingmoney (= lowMoney × rate / 100)
4. 월가용소득 = 월평균 소득 - 생계비 - 추가생계비 - 양육비 - 외부회생위원보수
5. 변제기간 결정:
   - repaymentperiodsetting 1~6 규칙 적용
   - 또는 forcingrepaymentmonthoption 1~3 수동 재설정
6. 청산가치 계산:
   - 재산 총액 × 라이프니츠 계수 할인
   - except_leibniz 시 할인 없음
7. 월변제액 = max(
     월가용소득 (청산가치 / 변제개월수) — 중 큰 값  // 청산가치 보장 원칙
   )
8. 라운딩·마지막달 보정 (규칙 미확인)
9. 서버 리포트 템플릿이 월별 변제표 생성
```

**⚠️ 현재 공란 / 미확인**:
- 월가용소득 계산에서 세금·4대보험 공제가 포함되는지
- 청산가치 보장 원칙 적용 방식 (월별? 누적?)
- 라운딩: 원단위 ceil/round/floor
- 마지막 달 보정: 잔액 전부 / 비례 배분
- 이자 포함·제외 스위치가 `repaymentperiodsetting`과 어떻게 연결되는지

## 8. 결론 (중간)

colaw의 변제계획안 자동작성은 단순한 "가용소득 ÷ 개월수" 공식이 아니라:
- **6가지 변제기간 규칙**
- **증액 가능한 생계비**
- **라이프니츠 현가계수 기반 청산가치**
- **추가생계비·양육비·보수율 별도 공제**
- **이자 포함/제외 스위치**
- **변제율 표기 세밀 제어**

를 포함한 **복합 의사결정 엔진**이다. VS 현 상태로 "공식이 같다"고 단정할 수 없으며, 위 체크리스트 16개 항목을 모두 VS에 구현한 후에야 원 단위 일치가 가능하다.

### 9.6 미확정 채권 처리 (재해석 — 2026-04-07)

**이전 가설 폐기**: 채권별 `unsettlementbond` 체크박스는 10건 모두 `false`였음. 단순 per-creditor flag 아님.

**실제 메커니즘** (colaw `popupRescureCreditorList` 핸들러 소스 분석):

colaw는 미확정을 두 개의 **상호 배타적** 경로로 저장한다.

| 경로 | 트리거 버튼 | 팝업 | 용도 |
| --- | --- | --- | --- |
| A. 별제권부(담보부) | `.layer-attached-documents` | `popupAttachedDocumentsList` | 담보부 채권의 부족액 (부속서류 1~4) |
| B. 기타 미확정 | `.layer-unconfirmed-bond` | `popupUnConfirmedBondView` | 신탁재산 등 법률상 미확정 |

핸들러 주석 원문: *"부속서류1,2,3,4 선택 시 기타미확정채권(신탁재산등) 존재 체크"*, *"기타미확정채권 선택 시 부속서류1,2,3,4 존재 체크"* — 한 채권자는 A 또는 B 중 하나만 가질 수 있음. 존재 체크 엔드포인트는 `/rescureManage/ajaxIsExistResurAttachedDocUndeterminedBonds`.

**김한경 13,155,499원은 A 경로 (별제권부)**:
- 제이비우리캐피탈(주) 자동차담보대출금 총액 23,835,499원
- `frm_bond_calculate` 수준에 `dambosum=10,680,000` / `nodambosum=56,714,484` 저장됨 (케이스 전체 합계)
- 담보평가액 10,680,000 = B_29 리포트의 담보부 완제 원금과 정확히 일치
- **부족액 = 23,835,499 − 10,680,000 = 13,155,499** (원 단위 일치 확인)
- 즉 "미확정 채권"이라 표시되지만 실제는 **부속서류로 선언된 별제권부의 담보 부족분**이 무담보로 편입된 금액

**colaw 채권별 담보평가액 필드 캡처 블로커**:
- `popupAttachedDocumentsList` 직접 POST/iframe 로드 모두 세션 리다이렉트됨(`/myPage/pleaseLoginAgain`). 존재 체크 API는 server 500.
- 원인: 해당 팝업은 opener chain + 특정 server-side 상태 의존. migration 단계에서 정상 UI 플로우로 모달을 열어 필드명(`dambosum_each`, `appraisalamount` 등 추정) 확정 필요.
- 블로커 영향: VS 스키마 설계는 의미 기준(`secured_collateral_value`)으로 선행 가능. colaw→VS 마이그레이션 스크립트의 채권별 담보평가 추출은 필드명 확정 후 구현.

**VS 이식 함의**:
- 단순 `confirmation_status` 컬럼 추가 방식 **폐기**
- `rehabilitation_creditors`에 `is_secured boolean`, `secured_collateral_value numeric` 추가
- (희귀) 기타 미확정용 `is_other_unconfirmed boolean` 별도 플래그
- 월변제 분리 로직: `unconfirmedPortion = max(0, totalClaim − securedCollateralValue)` (별제권부), 또는 `totalClaim`(기타 미확정)
- VS `creditor-mismatch` 70건은 이 구조 미구현 → schedule-generator가 확정/미확정 분리 출력을 못하기 때문일 가능성이 높음
