# 라이프니츠 계수 K vs L 충돌 분석

**작성일**: 2026-04-08
**근거 케이스**: 김한경 (cs=5640948, anatomy §9.3/§9.4)
**상태**: 결정 보류 — 추가 케이스 확보 후 P1-10에서 결정

---

## 1. 충돌 개요

P0-1에서 라이프니츠 36개월 계수를 **33.7719**(공표 4자리 표값)로 채택했음. 그러나 김한경 케이스의 K값(20,212,548)은 이 계수로는 재현되지 않음. 33.7702로만 K가 정확히 떨어짐.

| 계수 | L 검증 | K 검증 |
|---|---|---|
| **33.7719** (P0-1 채택) | ✅ 정확 | ❌ −100원 |
| **33.7702** (anatomy §9.4 초안) | ❌ −95원 | ✅ 정확 |

두 계수가 양립 불가. 한쪽은 L, 다른 쪽은 K만 맞춤.

---

## 2. 검산

### 2-1. 33.7719 (현재 P0-1 채택)

```
L = round(561,457 × 33.7719)
  = round(18,961,469.66)
  = 18,961,470 ✓ (anatomy §9.4 일치)

K_recomputed = round(L × 36 / 33.7719)
             = round(18,961,470 × 36 / 33.7719)
             = round(682,612,920 / 33.7719)
             = round(20,212,447.9)
             = 20,212,448 ✗ (anatomy §9.4 K=20,212,548과 -100원 차이)
```

### 2-2. 33.7702 (anatomy 초안)

```
L = round(561,457 × 33.7702)
  = round(18,961,374.65)
  = 18,961,375 ✗ (anatomy L=18,961,470과 -95원 차이)

K_recomputed = round(L × 36 / 33.7702)
             = round(18,961,470 × 36 / 33.7702)
             = round(682,612,920 / 33.7702)
             = round(20,212,548.28)
             = 20,212,548 ✓ (anatomy K 일치)
```

---

## 3. 가설

### 가설 A — 두 상수 분리 사용
colaw 내부에서 L과 K를 별도 계수로 계산.
- L = `round(payment × 33.7719)`
- K = `round(L × 36 / 33.7702)`

장점: 양쪽 모두 정확 재현 가능. anatomy §9.4의 두 값(`33.7719`, `33.7702`)이 어디선가 동시에 존재한다는 가설.

단점: 동일 변제기간에 두 상수를 쓰는 이유가 불명확. 실무 관행 또는 버그 가능성.

### 가설 B — K가 라이프니츠와 무관
K = `monthlyAvailable × months + 보정` 또는 다른 산출.
- 561,457 × 36 = 20,212,452
- 20,212,548 − 20,212,452 = 96 (보정값)

96원의 출처가 라이프니츠가 아닐 가능성. 회차당 4원 × 24회 = 96 분산. P1-9 `buildAdjustedSchedule` 동작과 일치하지만, 96의 산출 근거는 미해명.

장점: P0-1의 33.7719 결정을 유지 가능.

단점: K의 산출 공식 자체가 미확인. 다른 케이스에서도 동일 패턴인지 검증 필요.

### 가설 C — 단일 계수 + 별도 보정 정책
실제 colaw 계수는 33.7702 (anatomy 초안 정답). L의 95원 차이는 별도 round-up 또는 보정 적용.

검증: `561,457 × 33.7702 = 18,961,374.65 → ceil = 18,961,375`. 95원 모자람. **ceil로도 18,961,470은 안 나옴**. 가설 C는 즉시 배제.

---

## 4. 결정 보류 사유

한 케이스(김한경)만으로는 가설 A와 B를 구분할 수 없음. 필요한 추가 데이터:

1. **다른 36개월 케이스 1건 이상** — K, L 동시 캡처 → 가설 A의 두 상수 패턴이 일관되는지 검증
2. **48개월 또는 60개월 케이스** — 해당 기간 라이프니츠 계수 4자리 vs K 역산값 확인
3. **colaw 프론트 JS 소스** — `presentValue` / `K_compute` 함수 직접 추적

---

## 5. 현재 처리

P0-1의 `33.7719`는 **유지**. 사유:
- 공표 4자리 표값과 일치 (대법원 양식 기준)
- L은 정확 재현 (presentValue 함수의 1차 출력)
- K의 +100원 차이는 P1-9 `buildAdjustedSchedule`의 `totalTarget` 외부 주입으로 흡수 가능

P1-9에서 `totalTarget`을 외부에서 받아 회차별 분산하므로, K의 출처가 라이프니츠 역산이든 별도 공식이든 동일하게 처리됨. 즉 P1-9는 본 충돌과 무관하게 정확.

## 7. 영향 범위 (현 시점)

| 모듈 | 영향 |
|---|---|
| `leibniz.ts` `presentValue` | 영향 없음 (L 출력 정확) |
| `repayment-period.ts` `decideRepaymentPeriod` | 영향 없음 (PV 비교) |
| `period-setting.ts` `decidePeriodSetting` | `totalScheduled = monthly × months` 단순 산출 → K와 +96원 차이 |
| `rounding.ts` `buildAdjustedSchedule` | `totalTarget` 외부 주입 가능, 영향 없음 |
| `document-generator.ts` 변제계획안 표 | K 표기 시 +100원 차이 가능 |

P1-9가 외부 주입 경로를 열어둔 덕분에, K 결정 후 위에 한 줄 연결만 하면 됨.
