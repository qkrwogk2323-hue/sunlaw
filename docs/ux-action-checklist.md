# UX Action Checklist

이 문서는 버튼, 폼, 승인, 요청, 삭제, 전환 같은 사용자 액션을 설계할 때 빠르게 점검하는 기준입니다.

기존의 1~10 나열형 체크리스트를 실제 사용자 흐름에 맞춰 4단계로 다시 묶었습니다.

## 왜 4단계로 보나

- 나열형 목록보다 사용자 흐름과 구현 순서를 함께 보기 쉽습니다.
- 기능을 만들 때 "이 액션은 어느 단계까지 챙겨야 하나"를 바로 판단할 수 있습니다.
- 서버 액션, 폼 상태, 권한 가드, 알림 문구를 같은 기준으로 검토할 수 있습니다.

## Phase 0. 행동 유발

### 1. 사용자 행동 트리거

- 버튼, 링크, 입력창, CTA가 무엇을 하는지 즉시 이해돼야 합니다.
- 액션 이름은 결과 중심으로 작성합니다.
- 예시: `저장`, `삭제`, `승인`보다 `권한 저장`, `조직 삭제`, `승인하고 조직 생성`이 낫습니다.

### 2. 사전 안내

- 실행 전 필요한 전제 조건, 영향 범위, 되돌릴 수 있는지 여부를 미리 보여줍니다.
- 파괴적 액션은 특히 대상, 영향, 후속 상태를 짧게 알려야 합니다.

## Phase 1. 안전장치

### 3. 권한 및 조건 검증

- UI 노출 여부와 서버 권한 검증을 분리하지 말고 둘 다 챙깁니다.
- 버튼이 보여도 서버에서 다시 막아야 합니다.
- 필수 입력값, 조직 문맥, 멤버십, 상태 전이 가능 여부를 확인합니다.

### 4. 중간 확인 절차

- 삭제, 승인, 권한 변경, 외부 발송처럼 영향이 큰 액션은 확인 단계를 둡니다.
- 확인 문구에는 대상과 결과를 명시합니다.
- 단순 확인보다 체크박스, 대상명 재입력, 요약 카드가 더 안전한 경우가 많습니다.

## Phase 2. 실행 & 피드백

### 5. 처리 상태 표시

- 요청 전송 중, 저장 중, 승인 중 같은 진행 상태를 즉시 보여줍니다.
- 중복 제출을 막고, 버튼 비활성화 또는 `aria-busy`를 함께 적용합니다.

### 6. 결과 피드백

- 성공, 실패, 부분 성공을 구분해 알려줍니다.
- 실패 문구는 사용자가 다음에 무엇을 해야 하는지 포함해야 합니다.
- 예시: `저장 실패`보다 `저장 실패: 필수 입력값을 확인해 주세요.`가 낫습니다.

### 7. 상태 반영

- 액션 직후 목록, 상세, 뱃지, 개수, 최근 활동 등 관련 화면이 함께 바뀌어야 합니다.
- 성공 메시지만 보이고 실제 UI가 이전 상태면 신뢰가 떨어집니다.

## Phase 3. 후처리 & 신뢰

### 8. 취소 및 복구

- 되돌릴 수 있는 액션은 undo, 복원, 재시도, 임시보관 중 하나를 제공합니다.
- 완전 삭제만 제공하는 설계는 마지막 선택지여야 합니다.

### 9. 일관성

- 버튼 위치, 색상, 문구, 상태 표시, 에러 형식이 화면마다 달라지지 않게 맞춥니다.
- 같은 위험도의 액션은 같은 톤으로 보여야 합니다.

### 10. 시스템 메시지 체계

- 토스트, 인라인 에러, 빈 상태, 배너, 알림센터 메시지의 역할을 구분합니다.
- 무엇을 어디에 띄울지 기준이 없으면 중복 안내와 누락이 동시에 생깁니다.

## 추가로 꼭 필요한 5가지

이 다섯 항목은 액션 체크리스트만으로 빠지기 쉬운 운영 필수 항목입니다.

| 우선순위 | 항목 | 왜 필요한가 | 구현 기준 |
| --- | --- | --- | --- |
| 1 | 접근성 | 법적 대응, SEO, 키보드 사용성, 스크린리더 대응 | `aria-label`, `role`, `focus-visible`, 키보드 탐색, 대비 점검 |
| 2 | 마이크로 인터랙션 | 사용자가 클릭과 상태 변화를 즉시 체감 | hover, press, pending motion, 상태 전환 애니메이션 |
| 3 | 로깅 & 분석 | 이탈 구간과 오류 구간을 나중에 추적 가능 | analytics 이벤트, 서버 로그, 예외 수집 |
| 4 | 다국어 / 테마 대응 | 확장성, 운영 유연성, 브랜드 일관성 | locale 구조, 문구 리소스화, 라이트/다크 테마 전략 |
| 5 | 로드 상태 / 네트워크 에러 처리 | 느린 네트워크에서 사용자 불안과 중복 입력 방지 | skeleton, retry, 네트워크 오류 메시지, timeout 대응 |

## 구현 시 빠른 점검 질문

- 사용자는 이 버튼을 누르기 전에 결과를 예측할 수 있는가?
- 권한과 조건 검증이 UI와 서버 모두에 있는가?
- 진행 상태와 결과가 액션 직후 바로 보이는가?
- 실패했을 때 다시 시도하거나 복구할 수 있는가?
- 같은 종류의 액션이 다른 화면에서도 같은 방식으로 동작하는가?

## 운영 점수 기준

- 90점: 바로 개발 진행 가능. 핵심 액션 흐름은 정리돼 있음.
- 95점: 4단계 구조로 재정리되어 설계와 리뷰 속도가 빨라짐.
- 100점: 접근성, 분석, 네트워크 예외, 다국어/테마, 마이크로 인터랙션까지 반영됨.

## 이 레포에서의 사용법

- 새 폼이나 서버 액션을 추가할 때는 이 문서 기준으로 최소 Phase 0~2를 먼저 점검합니다.
- 삭제, 승인, 권한 변경, 외부 공유 기능은 Phase 3까지 포함해 검토합니다.
- PR 설명에는 필요한 경우 "어느 Phase를 반영했는지"를 짧게 남깁니다.
