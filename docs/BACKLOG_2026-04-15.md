## 1. PII_ENCRYPTION_KEY_BASE64 재암호화 프로젝트

**우선순위**: P1 (보안)
**소유**: 개발팀 + 보안 리뷰
**예상 기간**: 3~5일 (준비 1일 + 마이그레이션 2~3일 + 검증 1일)

### 배경
2026-04-15 보안 incident(`docs/SECURITY_INCIDENT_2026-04-15_SECRET_EXPOSURE.md`)에서 `PII_ENCRYPTION_KEY_BASE64`가 세션 로그에 평문 노출됨. 다른 6종 secret은 전부 회전됐으나 이 키는 기암호화된 PII 데이터가 있어 단순 교체 시 복호화 불가 → 유예.

현재 service_role 회전으로 즉시 위협은 차단됐지만, PII 키 단독 탈취 경로가 이론상 남아있어 계획적 교체 필요.

### 목표
구 키(`eB5dq2HS...`) 폐기 + 신규 키로 전체 PII 컬럼 재암호화 + 애플리케이션이 신규 키로만 복호화.

### 영향 범위
암호화 컬럼(`src/lib/pii.ts`의 `encryptString`/`decryptString` 호출처):
- `case_party_private_profiles.resident_number_ciphertext`
- `case_party_private_profiles.registration_number_ciphertext`
- `case_party_private_profiles.address_detail_ciphertext`
- 기타 `*_ciphertext` 컬럼 (쿼리로 전수 조사 필요)

호출처:
- `src/lib/actions/profile-actions.ts`
- `src/lib/actions/client-account-actions.ts`
- `src/lib/actions/case-actions.ts`
- `src/lib/actions/organization-actions.ts`
- `src/app/api/auth/general-signup/route.ts`
- `src/lib/queries/clients.ts`

### 선결 조건
- [ ] 암호화 컬럼 전수 조사 완료 (information_schema + grep `*_ciphertext`)
- [ ] 각 컬럼의 row count 파악 (재암호화 소요 시간 추정)
- [ ] staging에서 dual-key 모드 시나리오 성공 확인

### 세부 단계

**Phase 1 — Dual-key 도입 (배포 1차)**
1. `PII_ENCRYPTION_KEY_BASE64_NEW` 환경변수 추가 (신규 키)
2. `src/lib/pii.ts` 수정:
   - `encryptString`: `_NEW` 존재 시 신규 키로만 암호화 (key version prefix `v2:` 같은 걸로 구분)
   - `decryptString`: prefix에 따라 분기 — `v2:`면 NEW로, 그 외는 구 키로 복호화
3. 로컬·staging에서 기존 데이터 복호화 + 신규 저장 시 신규 키 암호화 확인
4. 프로덕션 배포 — 이 시점부터 신규 INSERT/UPDATE는 신규 키로 암호화

**Phase 2 — 백필 마이그레이션 (staging 먼저, 그다음 prod)**
1. 스크립트 작성: `scripts/reencrypt-pii.mjs`
   - service_role로 암호화 컬럼 순회
   - prefix 없는 row만 대상: 구 키 decrypt → 신규 키 encrypt → UPDATE
   - chunk size 500, 실패 시 멱등 재실행 가능하게
2. staging에서 먼저 실행, row count · 소요 시간 · 에러 샘플 확인
3. prod 실행 (업무시간 외 권장, lock 영향 확인)

**Phase 3 — 구 키 폐기 (배포 2차)**
1. 모든 row가 `v2:` prefix 붙었는지 검증 쿼리
2. `src/lib/pii.ts`에서 구 키 decrypt 경로 제거
3. `.env.local` / Vercel / GitHub secrets에서 `PII_ENCRYPTION_KEY_BASE64` 제거, `_NEW` → `PII_ENCRYPTION_KEY_BASE64`로 이름 정리
4. 프로덕션 배포

### 완료 기준
- [ ] Phase 1~3 전부 머지 + 프로덕션 배포 완료
- [ ] 전수 검증 쿼리로 `v2:` prefix 아닌 row 0건 확인
- [ ] `docs/SECURITY_INCIDENT_2026-04-15_SECRET_EXPOSURE.md`의 PII 유예 항목을 `[x]`로 체크 + 완료 일시 기록

### 롤백 계획
- Phase 1 배포에서 문제 발생 시: 구 `pii.ts` 되돌리기 (키 둘 다 유지하니 무해)
- Phase 2 백필 실패 시: 이미 재암호화된 row는 `v2:` prefix라 dual-key 모드가 알아서 처리. 스크립트 재실행으로 이어서 진행
- Phase 3 이후: 구 키 폐기됐으므로 비가역. Phase 2 검증이 100%여야 Phase 3 진입

## 3.원. 원본 설계 (archive)

아래는 2026-04-15 백로그 작성 시점의 원본 설계. 2026-04-16 완료 후 구현과 차이가 있으면 비교용으로만 참조.

### 배경
현재 staging migration 적용은 수동 + 자체 preprocessor 기반 (Supabase Management API `/v1/projects/:ref/database/query`).
- `AS $tag$` 뒤 세미콜론 보완, `CREATE TRIGGER` 앞 `DROP TRIGGER IF EXISTS` 삽입
- seed_data의 nested `$$` → `$cron_body$` 치환
- hotfix_008의 `COMMENT ON TABLE storage.buckets` 스킵

재현성은 있지만 자동화 안됨. PR 머지 → staging 자동 반영 파이프라인 없음.

### 목표
PR 머지 → staging DB 자동 migration 적용 → schema drift 자동 검증 → 문제시 자동 알림.

### 세부 단계
1. GitHub Actions workflow 추가: `.github/workflows/staging-deploy.yml`
   - trigger: main push + migration 파일 변경 시
   - step 1: `supabase/migrations/*.sql` 변경분 추출
   - step 2: preprocessor 실행 후 `/v1/projects/:ref/database/query` 호출
   - step 3: `pnpm check:schema-drift` with `STAGING_*` env
2. preprocessor를 `scripts/preprocess-migration.mjs`로 독립 (재사용 가능하게)
3. schema drift 감지 시 GitHub Issue 자동 생성 (repo maintainer에게 멘션)
4. staging 상태 대시보드 (선택) — `/api/admin/staging-health` endpoint, prod와 staging 핵심 메트릭 비교

### 완료 기준
- [ ] PR이 migration 포함 시 staging 자동 반영 확인
- [ ] schema drift 0건 유지
- [ ] 수동 `bash /tmp/rotate-*` 같은 one-shot 스크립트 없이 재현 가능
