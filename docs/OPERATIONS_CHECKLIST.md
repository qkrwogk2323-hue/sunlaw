## 🔧 섹션 1: 인프라 설정 (개발팀)

### 1-1. Sentry 에러 모니터링 (즉시 필요)
- [ ] [sentry.io](https://sentry.io) 접속 → 새 프로젝트 생성 (Platform: Next.js)
- [ ] DSN 복사 → Vercel 환경변수 `NEXT_PUBLIC_SENTRY_DSN` 추가 (Production + Preview)
- [ ] `.env.local`에 `NEXT_PUBLIC_SENTRY_DSN=https://xxx@xxx.ingest.sentry.io/xxx` 추가
- [ ] `NEXT_PUBLIC_APP_ENV=production` Vercel 환경변수 추가
- [ ] (선택) `SENTRY_AUTH_TOKEN` 추가 시 소스맵 업로드 활성화
- [ ] 배포 후 테스트: 로그인 후 `/api/auth/general-signup`에 잘못된 데이터 전송 → Sentry Issues 탭에서 에러 확인

> 현재 상태: 코드는 설치됨. DSN 없으면 아무 것도 캡처되지 않음.

---

### 1-2. PII 암호화 키 설정
- [ ] 32바이트 랜덤 키 생성: `openssl rand -base64 32`
- [ ] Vercel 환경변수 `PII_ENCRYPTION_KEY_BASE64` 에 출력값 추가
- [ ] **절대 Git에 커밋하지 말 것** — `.env.local` 전용
- [ ] 키 안전 보관: 1Password, AWS Secrets Manager, Vercel Encrypted 등

> 현재 상태: `PII_ENCRYPTION_KEY_BASE64` 없으면 주민등록번호 저장 시 서버 에러 발생

---

### 1-3. Supabase 환경 변수 확인
- [ ] `NEXT_PUBLIC_SUPABASE_URL` Vercel 환경변수에 추가됨
- [ ] `NEXT_PUBLIC_SUPABASE_ANON_KEY` 추가됨
- [ ] `SUPABASE_SERVICE_ROLE_KEY` 추가됨 (세션 강제 무효화 기능에 필요)
- [ ] `SUPABASE_SERVICE_ROLE_KEY`는 서버사이드 전용 — `NEXT_PUBLIC_` 접두사 절대 금지

---

### 1-4. 스테이징 환경 구성
- [ ] Supabase 프로젝트 신규 생성 (스테이징 전용)
- [ ] Vercel Preview 브랜치를 스테이징 Supabase에 연결
- [ ] 스테이징 전용 환경 변수 세트 Vercel에 등록
- [ ] 스테이징 DB에 시드 데이터 적용: `supabase db seed --file supabase/seeds/0001_demo_seed.sql`
- [ ] 스테이징에서만 기능 테스트 → 프로덕션 배포

> 현재 상태: 개발/프로덕션만 존재. 프로덕션에서 직접 테스트 중.

---

### 1-5. DB 백업 복구 테스트
- [ ] Supabase 대시보드 → Settings → Database → Backups 확인
- [ ] 최신 자동 백업 1개 선택 → "Restore to new project" 실행
- [ ] 복구된 프로젝트에서 주요 테이블(cases, profiles, organizations) 데이터 검증
- [ ] 복구 완료 시간 기록 (RTO 측정)
- [ ] **분기 1회** 반복 실행

> 현재 상태: 백업 존재 여부 불명. 복구 테스트 미실시.

---

### 1-6. 의존성 보안 감사
- [ ] CI `pnpm audit --audit-level=high` 활성화됨 ✅ (이미 완료)
- [ ] 월 1회 `pnpm audit` 결과 검토 → HIGH/CRITICAL 취약점 즉시 패치

## 🔒 섹션 4: 보안 운영 (운영팀)

### 4-1. 직원 퇴사 시 처리 절차
현재 시스템 기능: 팀 설정 → 구성원 → "세션 종료" 버튼으로 즉시 세션 무효화 가능.

**퇴사 처리 SOP (Standard Operating Procedure):**
- [ ] 운영팀이 아래 절차 문서화:
  1. 퇴사일 당일: 팀 설정 → 해당 구성원 "세션 종료" 클릭
  2. 즉시 조직 멤버십 "비활성화" 처리 (구성원 삭제 또는 상태 변경)
  3. 해당 직원이 사용하던 공유 계정(있다면) 비밀번호 변경
  4. Supabase 대시보드에서 해당 Auth 사용자 직접 비활성화 확인

---

### 4-2. 정기 보안 점검
- [ ] 월 1회: `pnpm audit` 실행 → 취약점 리포트 검토
- [ ] 분기 1회: DB 백업 복구 테스트 실행
- [ ] 연 1회: Supabase RLS 정책 전체 검토
- [ ] 연 1회: 환경변수 키 교체 (`PII_ENCRYPTION_KEY_BASE64` 등)

---

### 4-3. RLS 정책 실제 동작 검증
현재 상태: RLS 정책 작성됨. 실제 cross-org 쿼리 차단 테스트 미실시.

- [ ] 개발팀 + 운영팀: 스테이징 환경 구축 후 아래 시나리오 수동 테스트:
  1. 조직 A 계정으로 로그인
  2. 조직 B의 `case_id`를 직접 URL에 입력
  3. 접근 거부(403/404) 확인
  4. Supabase Admin → SQL Editor에서 cross-org 쿼리 직접 실행 후 빈 결과 확인
- [ ] pgTAP 또는 Supabase CLI 기반 자동화 테스트 작성 (개발팀 장기 과제)
