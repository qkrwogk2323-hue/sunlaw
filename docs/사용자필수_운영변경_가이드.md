# 사용자 필수 운영변경 가이드

이 문서는 사용자(운영 담당 포함)가 변경 시 반드시 확인해야 하는 항목을 빠르게 확인하기 위한 가이드입니다.

## 1) 변경 시 가장 먼저 확인할 것

1. 로그인/권한 관련 변경인지
2. 데이터 쓰기 경로(INSERT/UPDATE/DELETE) 변경인지
3. 감사로그(`/admin/audit`) 노출 기준 변경인지
4. 스키마(migration) 추가/변경인지

하나라도 해당되면 문서 동기화가 필수입니다.

## 2) 사용자전용 화면 변경 룰

대상: `/admin/audit` 기록보관함, 사용자에게 직접 노출되는 이력/에러 안내 화면

필수 규칙:
1. 화면 필터/문구/권한/조회 기준을 바꾸면 문서도 같은 PR에서 수정
2. 운영자가 "어떤 이벤트가 기록되는지"를 문서에서 바로 찾을 수 있어야 함
3. 문서 미반영 변경은 배포 보류

## 3) 기록보관함에 저장되는 트리거 기준

기본 저장소는 `audit.change_log`입니다.

저장 조건:
1. 대상 테이블에 `audit.capture_row_change()` 트리거가 붙어 있음
2. 실제로 `INSERT/UPDATE/DELETE`가 발생함
3. 권한/RLS로 조회가 차단되지 않음

주의:
- 앱 코드에서 넣는 `audit_logs`는 별도 로그이며 `/admin/audit`와 자동 통합되지 않습니다.

## 4) 마이그레이션 운영 원칙

1. `0001`부터 최신 번호까지는 "수정"이 아니라 "추가"로 관리합니다.
2. 새 migration 생성 시, 설명 문서도 즉시 갱신합니다.
3. 중복 번호/특이 이력(예: `0032` 중복)은 문서에 명시합니다.

참조 문서:
- `docs/마이그레이션_스키마_설명서_0001_0070.md`

## 5) 장애 발생 시 최소 점검 순서

1. 증상 재현
2. DB 트리거 존재 확인
3. 감사로그 실제 적재 여부 확인
4. 권한/RLS 확인
5. 앱 로그와 감사로그 소스 혼동 여부 확인
6. 핫픽스 전후 e2e/스모크 테스트 실행

## 6) 릴리즈 전 체크리스트 (사용자 영향 기준)

1. 사용자 로그인/권한 플로우 정상
2. 핵심 데이터 저장/수정/삭제 정상
3. `/admin/audit`에서 변경 이력 확인 가능
4. 스키마 설명 문서 최신 상태
5. 버그리포트/운영 가이드 최신 상태
