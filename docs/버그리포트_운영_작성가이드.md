# 버그리포트 운영 작성 가이드

이 문서는 운영/개발자가 같은 형식으로 버그를 기록하고, 누락 없이 추적하기 위한 표준입니다.

## 1) 기본 원칙

1. 재현 가능한 사실만 기록합니다.
2. "영향도(사용자/데이터/권한)"를 반드시 포함합니다.
3. 임시조치와 근본원인을 분리해서 기록합니다.
4. 수정 PR/커밋/배포를 동일 이슈에 연결합니다.

## 2) 심각도 기준

- `S0-Critical`: 로그인 불가, 결제/권한 전면 장애, 데이터 대량 손실
- `S1-High`: 핵심 기능 장애, 데이터 유실 위험 존재
- `S2-Medium`: 우회 가능한 기능 오류, 오탐/과탐, 부분 UX 장애
- `S3-Low`: 경미한 불일치, 문구/표시 문제

## 4) 트리거/감사로그 이슈 전용 체크

1. 대상 테이블에 `audit.capture_row_change()` 트리거가 있는지
2. 트리거가 `INSERT/UPDATE/DELETE` 전부를 커버하는지
3. `/admin/audit`에서 보이지 않으면 필터/권한/RLS를 확인했는지
4. 앱 이벤트 로그(`audit_logs`)와 감사로그(`audit.change_log`)를 혼동하지 않았는지

## 5) 배포 전 종료 조건

아래를 모두 충족해야 `Closed` 처리합니다.

1. 재현 시나리오가 더 이상 실패하지 않음
2. 관련 테스트(최소 1개 단위 또는 e2e) 추가/갱신됨
3. 관련 문서(가이드/스키마 설명서/운영 체크리스트) 업데이트 완료
4. 회귀 모니터링 항목이 설정됨
