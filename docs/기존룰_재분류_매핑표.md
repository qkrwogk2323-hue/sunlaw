# 기존룰 재분류 매핑표

이 문서는 `docs/PROJECT_RULES.md`와 `docs/UX_RULES.md`에 이미 존재하는 규칙을 새 분류 체계로 다시 읽기 쉽게 정리한 매핑표다.

중요 원칙:

1. 새 규칙을 만드는 문서가 아니다.
2. 기존 규칙 번호를 바꾸지 않는다.
3. `PROJECT_RULES.md`의 섹션 구조를 바꾸지 않는다.
4. 구현/버그/테스트는 반드시 이 매핑표를 통해 어느 규칙을 위반했는지 연결한다.

## 1. 분류 기준

### 1-1. 공통 제품 룰

- 디자인 시스템
- UX/UI
- 토스트, 모달, 배너, 로딩, 빈 상태
- 오류 표시
- 메시지 체계
- 접근성
- 공통 로그 표현
- 개발용어 금지

### 1-2. 플랫폼 조직 상세룰

- 조직 신청 관리
- 조직 승인/반려/삭제/복구
- 구독 권한 부여/회수
- 플랫폼 운영 알림
- 플랫폼 감사로그
- 플랫폼 전용 메뉴/기본 진입 경로

### 1-3. 사용자조직 공통 상세룰

- 사건
- 의뢰인
- 문서
- 일정
- 계약
- 비용
- 허브/협업
- 조직 내부 설정
- 사용자조직 공통 AI

### 1-4. 법무/법률사무소 상세룰

- 보정
- 회생/파산
- 법원 제출
- 기일/절차
- 법무 AI

### 1-5. 신용정보회사 상세룰

- 회수 활동
- 추심 운영
- 보수 산정
- 정산
- 신용정보회사 AI

### 1-6. 의뢰인 포털 상세룰

- 포털 권한
- 사건 진행 공유
- 요청/문서/청구/납부 확인
- 의뢰인 포털 AI

### 1-7. 개발코드 룰

- 권한 함수
- 단일 원천
- Server/Client 경계
- 검증 경계
- 트랜잭션, 마이그레이션, RLS
- 테스트/릴리즈 게이트
- 관측성
- 메뉴-기본 진입 경로 연결

### 1-8. 로그/오류/버그 방지 룰

- 감사로그
- 로그 싱크
- 설명 가능한 오류
- 로그 추적성
- 기능 완결성
- 재현 및 회귀 방지

## 4. 버그 원인 그룹과 룰 연결 방식

버그를 기록할 때는 반드시 아래 4개를 함께 적는다.

- `rule_id`
- `rule_scope`
- `root_cause_group`
- `regression_guard`

예시:

| 버그 유형 | 연결할 룰 | root_cause_group | regression_guard |
|---|---|---|---|
| 플랫폼 조직에 없는 대시보드가 기본 진입으로 뜸 | `PROJECT_RULES 5-3`, `5-13`, `3-4` | `navigation-single-source` | 메뉴 첫 주소와 기본 진입 경로 일치 테스트 |

## 5. 앞으로의 사용법

1. 기능을 고치기 전 먼저 이 문서에서 소속 규칙을 찾는다.
2. 버그를 기록할 때는 어떤 규칙을 위반했는지 `rule_id`로 남긴다.
3. 근본 원인은 `공통 제품`, `플랫폼`, `사용자조직`, `법무`, `신용정보회사`, `포털`, `개발코드`, `로그/오류` 중 어디서 발생했는지 같이 적는다.
4. 같은 `root_cause_group`에서 파생된 버그는 새 버그가 아니라 파생 버그로 연결한다.
5. 회귀 테스트 또는 체크리스트가 없으면 버그를 닫지 않는다.
