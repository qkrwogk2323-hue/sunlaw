### 폼 필드
```tsx
// ✅ 필수 입력란 — 빨간 * 표시 필수
<div className="space-y-1">
  <label htmlFor="name" className="text-sm font-medium">
    이름 <span className="text-red-500" aria-hidden="true">*</span>
  </label>
  <Input id="name" name="name" required aria-required="true" />
</div>
// 폼 상단에 안내문
<p className="text-xs text-slate-500"><span className="text-red-500">*</span> 필수 입력 항목입니다</p>
```
- `required` 필드 → `<span className="text-red-500">*</span>` 필수
- `label htmlFor` ↔ `input id` 반드시 연결
- `placeholder`는 label 대체 불가, 보조 안내만
- 폼 상단에 필수/선택 안내를 표시
- URL, 이메일, 사업자번호, 전화번호, 주민번호는 형식 힌트를 같이 표시
- raw Zod/JSON/영문 검증 에러를 그대로 노출하지 않음
- 개발언어, 내부 코드 용어, 영어 시스템 문구, 개발자 설명, 추상적인 표현을 UI에 그대로 노출하지 않음
- 같은 의미의 상태 카드와 안내 박스는 크기와 정렬을 맞추고 글자가 깨지지 않게 구성
- 플랫폼 전용 기능이 일반 사용자조직이나 의뢰인 화면에 보이거나 실행되면 버그로 간주하고 플랫폼 관리조직 로그로 기록
- 본인이 하지 않은 staged 변경은 절대 건드리지 않음. 푸시가 막히면 현재 상태를 먼저 보고하고 브랜치/로컬 보존 여부를 사용자에게 확인

### Soft Delete (UX #8 강제)
```ts
// ❌ 절대 금지 — 즉시 hard delete
await supabase.from('table').delete().eq('id', id);

// ✅ 필수 — lifecycle_status 방식
await supabase.from('table')
  .update({ lifecycle_status: 'soft_deleted', updated_by: auth.user.id })
  .eq('id', id);

// ✅ 필수 — deleted_at 방식 (lifecycle_status 없는 테이블)
await supabase.from('table')
  .update({ deleted_at: new Date().toISOString() })
  .eq('id', id);
```
- 목록 쿼리: `.neq('lifecycle_status', 'soft_deleted')` 또는 `.is('deleted_at', null)` 필수
- 삭제 후 반드시 `undo()` 토스트 + "보관함에서 복구 가능" 안내
- Trash UI(`?tab=trash` 또는 `/trash`): 복구 버튼 + 관리자만 영구삭제 가능
- 내부 롤백 삭제는 `PROJECT_RULES.md 2-5-1` 보상 삭제 예외를 따른다

### 🔴 Migration 규칙 (절대 준수)
```
supabase/migrations/ 구조:
  001 extensions_and_schemas   007 insolvency_shared
  002 enums                    008 rehabilitation
  003 core_tables              009 functions_and_triggers
  004 platform_governance      010 rls_policies
  005 collaboration            011 indexes
  006 billing                  012 seed_data
```
- **새 migration 파일 추가 금지** — 기존 12개 파일 안에서 수정
- 컬럼 추가/변경 → 해당 도메인의 기존 CREATE TABLE 문을 직접 수정
- 함수 추가/변경 → `009_functions_and_triggers.sql` 수정
- RLS 추가/변경 → `010_rls_policies.sql` 수정
- 인덱스 추가 → `011_indexes.sql` 수정
- 시드 데이터 → `012_seed_data.sql` 수정
- 새 테이블이 필요하면 가장 가까운 도메인 파일(003~008)의 끝에 추가
- **배치 규칙**: 테이블 파일(003~008)에는 `CREATE TABLE` + 제약조건(constraint) + 코멘트만 둔다
  - `ENABLE ROW LEVEL SECURITY` + `CREATE POLICY` → `010_rls_policies.sql`에 집중
  - `CREATE INDEX` → `011_indexes.sql`에 집중
  - 테이블 파일에 인라인 RLS/인덱스를 넣지 말 것 — 대신 `-- NOTE: indexes → 011, RLS → 010` 코멘트 추가
- **절대 하지 말 것**: `0098_xxx.sql` 같은 증분 migration 파일 생성
- **이유**: 97개 → 12개로 squash한 구조. 다시 늘리면 안 됨
- 새 도메인 파일(013 이상)이 정말 필요하면 **사용자 승인** 후에만 추가. 기존 12개로 안 되는 이유를 먼저 설명할 것

### 🔴 올림/버림 규칙 (가이드 p.12-17)

```
(E) 월변제예정액 (채권자별) → Math.ceil  (올림)
(P) 재산처분 변제액 (채권자별) → Math.ceil  (올림)
(O) 변제투입예정액          → Math.ceil  (올림)
(L) 현재가치 (라이프니쯔)    → Math.floor (버림)
④ 회생위원 보수             → Math.round (반올림)
변제율(%)                   → Math.round (반올림)
```

올림 결과, (H)월변제예정액 합계가 (A)월가용소득보다 약간 많아지는 것이 **정상**.

### 가용소득 공식

```
③ 월 가용소득 = ① 월 수입 - ② 조정된 생계비
④ 회생위원 보수:
    채무총액 ≤ 2억 AND 급여소득자 → 법원사무관 → 0원
    그 외 → 외부 위원 → ③의 1% (반올림)
⑤ 월 실제 가용소득 = ③ - ④
```

생계비 = 기준중위소득 × 60% (원칙). `living_cost_rate` 디폴트 100 (= 60% 그대로).
추가 생계비: 주거비, 미성년자 교육비, 의료비 (법원 승인 필요).

### 안분 공식

```
(E) = (A)월실제가용소득 × { (D)해당채권 / (G)총채권 }  [올림]
```
안분 기준: **원금** 원칙. 이자도 변제 가능하면 원금+이자 합계.

### D5110 vs D5111

```
현재가치(가용소득 총변제) > 청산가치 → D5110 (가용소득만)
현재가치(가용소득 총변제) ≤ 청산가치 → D5111 (가용소득+재산처분)
```

D5111 투입액: `(O) = {(J)청산가치 - (L)현재가치} × 승수` [올림]
승수: 1년이내=1.3, 2년이내=1.5. 외부 위원 시 1% 차감.

60개월 = 3 + 50.6433 = 53.6433

### 채권자 가지번호 규칙 (처리지침 §5)

- 보증채무/연대보증 → 주채무자 채권번호의 **하위 번호** (예: 1-1, 1-2, 1-3)
- 가지번호는 주채무자 채권과 **같은 분류 그룹** 내에서만 부여해야 한다
- 동일 채권에 보증인이 복수이면 보증인 수만큼 가지번호를 연속 부여해야 한다
- 가지번호가 있는 채권의 `bond_number`는 주번호.부번호 형식으로 표시해야 한다 (예: `3-1`)

### 산정기준일 (처리지침 §3, 가이드 p.6)

- 채권자목록의 **산정기준일**(= 채권 현재액 산정 시점)을 반드시 표기해야 한다
- 산정기준일은 원칙적으로 **개인회생 개시신청일**로 한다
- 이자·지연손해금은 산정기준일 **전날**까지만 계산해야 한다
- D5106 채권자목록 상단에 `산정기준일: YYYY. MM. DD.` 형태로 명기해야 한다

### 변제개시일 규칙 (처리지침 §7③, §8④, §9①)

- 변제개시일 = 변제계획안 제출일 + 60~90일 내의 일정한 날로 정해야 한다
- 변제개시일이 확정되지 않은 경우 `변제개시일`이라는 문구로 대체 표기해야 한다
- 임치기간은 변제기간에 산입해야 한다 (처리지침 §8④)
- 변제계획안 제1항에 변제개시일을 명시해야 한다 (처리지침 §9①)

### 채권자목록 요약표 구조 (법원서식 D5106)

- 상단: **2분류** rowspan (담보부채권 / 무담보채권)
- 하단: **3분류** (별제권부 / 일반 확정 / 미확정)
- 요약표 열: 채권자 수, 원금 합계, 이자 합계, 채권현재액 합계
- 채권현재액 총합계 = 담보부 소계 + 무담보 소계
- 요약표는 채권자목록 본문 **앞**에 위치해야 한다

### 별제권 테이블 구조 (법원서식 D5106 부속서류)

- 5컬럼: **채권번호 / 담보종류 / 순위 / 목적물 / 채권최고액**
- 담보종류: 근저당권, 질권, 유치권, 양도담보, 가등기담보 중 택1
- 순위: 1순위, 2순위, 3순위 등 숫자로 기재
- 목적물: 부동산 → 소재지+면적, 자동차 → 차종+연식+등록번호
- 채권최고액: 설정 채권최고액 (원금과 다를 수 있음)

### 별제권 담보 환가비율 기본값 (가이드 §5, 실무)

```
부동산      → 70%   (감정가 대비 할인, REHABILITATION_LAW_RULES.md §6-1)
자동차      → 50%   (중고차 시세 대비, 감가상각·경매 낙찰률 반영, 법원 실무)
임차보증금  → 100%  (확정일자 보증금 = 액면가)
예금        → 100%  (잔액 = 액면가)
보험        → 100%  (해약환급금 = 액면가)
설비        → 70%   (사업용 설비·재고·비품)
채권        → 70%   (대여금·매출금 채권)
기타        → 70%   (기본값)
```
- `getDefaultValuationRate()` (`secured-allocation.ts`)에서 유형별 기본값을 반환한다
- 자동차 50%는 `getDefaultValuationRate()`에 `case '자동차': return 50` 으로 반영됨
- 환가비율은 사용자가 변경할 수 있으나, 위 기본값을 디폴트로 제공해야 한다
- 공식: `배분가능액 = 시장가치(market_value) × 환가비율(valuation_rate) / 100` (Math.round)

### 변제계획안 본문 10개 항목 (법원서식 D5110/D5111, 가이드 Ⅰ장)

변제계획안 본문은 법원 전산양식 D5110(가용소득만) 또는 D5111(가용소득+재산처분)에 따라 10개 항목으로 구성한다. 각 항목의 정식 제목과 필수 기재사항은 다음과 같다.

**1. 변제기간** (가이드 p.2, 처리지침 §8)
- `[YYYY]년 [MM]월 [DD]일부터 [YYYY]년 [MM]월 [DD]일까지 [N]개월간` 형식으로 기재해야 한다
- 변제시작일 = 변제계획안 제출일로부터 60일 후 90일 내의 일정한 날 (처리지침 §7③)
- 변제종료일 = 변제시작일 + 변제기간(개월)의 마지막 날
- 사용자 수정 가능 (날짜, 기간)

**2. 변제에 제공되는 소득 또는 재산** (가이드 p.3-5)
- **가. 소득**: (1) 수입, (2) 생계비, (3) 가용소득을 표로 기재해야 한다
  - 급여소득자 → 직장명 + 월 평균 수입 기재
  - 영업소득자 → 사업체명 + 월 평균 수입 기재
- **나. 재산**: D5110/D5111에 따라 분기
  - D5110 → `해당 없음` 체크, 재산처분 불필요
  - D5111 → `해당 있음` 체크 + 처분대상 재산(소재지/지목/면적 등), 변제기한(`인가일부터 1년 내` 또는 `2년 내`), 변제투입예정액, 회생위원 보수, 실제 변제투입예정액을 기재해야 한다

**3. 개인회생재단채권에 대한 변제** (가이드 p.5-6, 법 §583)
- `해당 있음` / `해당 없음` 체크
- **가. 회생위원의 보수 및 비용**: 법원사무관 선임 → 해당 없음, 외부 위원 선임 → 인가 전·후 보수란 모두 기재해야 한다
- **나. 기타 개인회생재단채권**: 법 §583 각호(원천징수 조세, 부가가치세, 임금·퇴직금 등) 해당 시 채권자명, 채권현재액, 채권발생원인, 변제기, 변제방법을 기재해야 한다
- 변제방법: 특별한 사정 없으면 양식 문구 그대로 사용

**4. 일반의 우선권 있는 개인회생채권에 대한 변제** (가이드 p.7)
- `해당 있음` / `해당 없음` 체크
- 해당 시: 채권자명, 채권현재액, 채권발생원인(국세/지방세/건강보험료 등 + 우선권 근거), 변제기, 변제방법을 기재해야 한다
- 변제방법: 특별한 사정 없으면 양식 문구 그대로 사용

**5. 별제권부 채권 및 이에 준하는 채권의 처리** (가이드 p.7-8)
- `해당 있음` / `해당 없음` 체크
- **가. 채권의 내용**: ①채권현재액(원금), ②채권현재액(이자), ③별제권행사 등으로 변제가 예상되는 채권액, ④별제권행사 등으로도 변제받을 수 없을 채권액을 기재해야 한다
- 별제권 등의 내용 및 목적물: 설정일, 원금, 이자율, 채권최고액, 환가예상액, 목적물 소재를 기재해야 한다
- **나. 변제방법**: (1) ③은 별제권행사로 변제, 가용소득/재산처분 대상에서 제외. (2) ④는 미확정채권으로 유보. (3) ④ 초과 확정 시 변제계획안 변경 절차로 변제

**6. 일반 개인회생채권에 대한 변제** (가이드 p.8-9)
- **가. 가용소득에 의한 변제**:
  - (1) 월 변제예정(유보)액 및 총 변제예정(유보)액: 원칙적으로 원금 기준 안분, `[ ]` 안에 `[원금]` 기재
  - (2) 변제방법: (가) 기간 및 횟수, (나) 변제월 및 변제일 — 매월 동일한 변제일에 변제하는 것이 원칙
  - ①항: 최초 변제일부터 변제계획인가일 직전 변제일까지 적립금의 변제방법
  - ②항: 인가일 직후 최초 도래 변제일부터 마지막 변제일까지의 변제방법
- **나. 재산의 처분에 의한 변제** (D5111만):
  - (1) 변제투입예정액 및 총 변제예정(유보)액: 원금 기준 안분, `[ ]` 안에 `[원금]` 기재
  - (2) 변제방법: (가)항에 재산처분 변제기한 기재 — 원칙 `인가일부터 1년 내`, 곤란 시 `2년 내`
  - (3) 강제집행 등의 효력: 특별한 사정 없으면 양식 문구 그대로 사용

**7. 미확정 개인회생채권에 대한 조치** (가이드 p.9)
- `해당 있음` / `해당 없음` 체크
- **가. 변제금액의 유보**: 특별한 사정 없으면 양식 문구 그대로 사용
- **나. 미확정 개인회생채권에 대한 변제**: 양식 문구 사용, 모든 `[ ]` 안에 `[원금]` 기재해야 한다

**8. 변제금원의 회생위원에 대한 임치 및 지급** (가이드 p.9)
- ① "위 `[ ]`항에 의하여"의 `[ ]`에 해당 항 번호를 기재해야 한다 — 3항~5항 및 7항 중 해당 있는 항, 6항은 항상 포함
- ② 개인회생위원의 예금계좌: 신청 시 공란, 추후 보완

**9. 면책의 범위 및 효력발생시기** (가이드 p.9, 법 §624)
- 특별한 사정 없으면 양식 문구 그대로 사용해야 한다
- 사용자 수정 가능하나 권장하지 않음

**10. 기타사항** (가이드 p.9)
- `해당 있음` / `해당 없음` 체크
- 해당 시 그 내용을 자유 기재
- 사용자 수정 가능

> `PLAN_SECTION_LABELS`와 `document-generator.ts`의 `generateRepaymentPlan()` 모두 위 법원서식 제목을 사용한다.

### 재산목록 카테고리 및 특수 계산 (법원서식 D5101)

14개 카테고리를 다음 순서로 기재해야 한다:
```
1.현금  2.예금  3.보험  4.자동차  5.임차보증금  6.부동산
7.사업용설비/재고/비품  8.대여금채권  9.매출금채권  10.예상퇴직금
11.(가)압류적립금  12.공탁금  13.기타  14.면제재산(주거용/생계비)
```

특수 계산 규칙:
- **예금/보험**: 각각 250만원 공제 가능 (법 §383①). 보험은 **보장성보험만** 공제 대상
- **예상퇴직금**: **1/2만** 청산가치에 반영 (나머지 1/2은 압류금지재산)
- **면제재산** (exempt_housing, exempt_living): 청산가치 **0원** 처리 (법 §383②)
- **임차보증금**: 주택임대차보호법 §8 우선변제 범위 내 금액은 압류 불가 → 청산가치 제외

압류불능재산 상세 분류 (법 §383, D5108/D5109):
- **D5108 개인회생재단에 속하지 않는 재산**: 법 §580③, §383①에 따른 압류금지재산을 기재해야 한다. 각 항목에 압류금지 근거 조문(`exemption_statute`)을 명시해야 한다
- **D5109 면제재산 결정신청**: 2가지 유형으로 구분해야 한다
  - `housing_deposit`: 주거용 건물 임차보증금 면제 (법 §383②) — 임대차계약서, 주민등록등본 첨부
  - `living_expense`: 6개월간 생계비에 사용할 특정재산 면제 (법 §383②)
- 청산가치 반영 방법:
  - 일반 재산: 전액 반영
  - 예금·보험(보장성): `max(0, 금액 - 공제액)` (공제 상한 250만원)
  - 예상퇴직금: `금액 / 2` (1/2만 반영, 반올림)
  - 면제재산(주거용/생계비): **0원** (청산가치 미반영)
  - 우선변제 임차보증금: **0원** (압류 불가 범위 전액 제외)

### 크로스 탭 정합성 검증 (실무 요건)

다음 데이터 간 정합성을 검증하고 불일치 시 경고해야 한다:

1. **채권자↔재산**: 별제권 담보물건(`secured_properties`)의 `property_type`이
   재산목록(`properties`)의 `category`에 대응 항목이 있어야 한다
2. **소득↔변제계획**: 소득·생계비 탭의 가용소득이 변제계획 탭의 월 변제액 기준과 일치해야 한다
3. **가족↔부양가족수**: 가족 탭의 `is_dependent=true` 인원수와 생계비 산정 시 가구수가 일치해야 한다
4. **채권자 합계↔요약표**: 채권자목록의 원금·이자 합계가 요약표의 수치와 일치해야 한다

경고일 뿐 저장/제출을 차단하지 않는다 (사용자 판단에 맡김).

### 채무 한도 자격요건 (법 §579)

```
담보부 채무 ≥ 15억원 → 개인회생 부적격
무담보 채무 ≥ 10억원 → 개인회생 부적격
```
자격요건 미충족 시 명확한 경고를 표시해야 한다.

### 미확정 채권 처리 (가이드 p.9 §7)

- 채권의 존재·액수에 다툼이 있어 확정되지 않은 채권은 **미확정**으로 처리해야 한다
- 변제금액은 확정 시까지 **유보**해야 한다
- 모든 `[ ]` 안에는 `[원금]`이라고 기재해야 한다
- 확정 후 유보액을 배분하거나 반환해야 한다
- 안분 계산 시 확정채권과 미확정채권은 **별도 산정 후 합산**해야 한다

### 마지막 채권자 잔여 보정 (실무)

- 올림(Math.ceil) 적용 시 채권자별 배분 합계가 총액을 초과할 수 있다
- **마지막 채권자**가 잔여분을 흡수하여 총액과 정확히 일치시켜야 한다
- 이 규칙은 월변제예정액 배분과 재산처분 배분 모두에 적용해야 한다

### 변제율 표시 규칙 (가이드 §2)

```
변제율(%) = (총변제액 / 무담보원금) × 100
```
- 소수점 이하 반올림 (Math.round)
- 무담보원금 = 일반 무담보 원금 + 별제권 부족분 (담보 초과 미회수 원금)
- 무담보원금이 0이면 총 원금을 분모로 사용해야 한다

### 생계비 산정 및 상한 (처리지침 §7②)

- 생계비 = 기준중위소득 × 60%가 원칙이며, 특별한 사정이 있는 경우 적절히 증감할 수 있다
- 60% 미만으로 정하는 것도 가능하다
- 60% 초과 시 **추가생계비 항목으로 사유를 기재**해야 한다 (법원 승인 필요)
- 추가생계비 인정 항목 (가이드 p.37 각주16, 서울회생법원 2021.2.15 기준): 주거비, 미성년자 교육비, 의료비
- 추가생계비는 **지속적·객관적 지출**이어야 하며, 소명자료 제출이 필수이다
- 배우자의 피부양자 인정 여부와 추가 생계비(주거비)는 법원(회생위원)의 조사결과에 따라 인정되지 않거나 감액될 수 있다

### D5106 채권자목록 전체 구조 (법원서식 D5106)

채권자목록은 다음 순서로 구성해야 한다:
1. **요약표**: 담보부/무담보 2분류 × 별제권부/일반확정/미확정 3분류 표
2. **본문**: 채권자별 상세 기재 (채권번호, 채권자명, 채권의원인, 원금, 이자, 산정근거, 주소, 부속서류 여부)
3. **부속서류 1**: 별제권부 채권 및 이에 준하는 채권의 내역 (5컬럼: 채권번호/담보종류/순위/목적물/채권최고액)
4. **부속서류 2**: 다툼이 예상되는 채권의 내역 (미확정채권 — 별제권 부족액, 소송 중 채권 등)

- 각 채권자 행에 `부속서류 □ (번호)` 체크란이 있어야 한다
- 미확정채권(별제권 부족액 포함)은 부속서류 2에 기재해야 한다

### 변제계획안 인가 후 변경 (법 §619)

- 변제계획 인가 후 변경이 필요한 경우, 채무자는 변제계획 변경안을 제출해야 한다 (법 §619)
- 제출 서류: 변제계획 변경안 제출서(D5122), 변제계획 수정안 제출서(D5121)
- 변경 사유: 소득 변동, 부양가족 변경, 채권 확정/변경 등
- 법원은 변경안에 대해 인가 여부를 결정한다
- 처리지침에는 변경 절차에 대한 별도 조항이 없으나, 법원서식 D5121/D5122가 존재한다

### 전자소송 제출 규칙 (전자소송규칙 §27②⑥)

- 개인회생사건은 **전자기록사건**이다 — 전자소송 동의 여부와 관계없이 강제 (전자소송규칙 §27②⑥)
- D5100 시리즈 서식 목록:
  - D5100: 개인회생절차 개시신청서

  - D5103: 수입 및 지출에 관한 목록
  - D5105: 진술서
  - D5106: 개인회생채권자목록
  - D5108: 개인회생재단에 속하지 않는 재산목록 제출서
  - D5109: 면제재산 결정신청서
  - D5110: 변제계획안 (가용소득만)
  - D5111: 변제계획안 (가용소득+재산처분)
  - D5112: 변제계획안 간이양식
  - D5114: 금지명령 신청서

### D5103 수입/지출 기재 규칙 (처리지침 §7①, 가이드 p.3)

- **월 평균 수입 계산**: 연간 환산 합계 / 12 (소수점 이하 올림, Math.ceil)
- **기간 환산 승수**: 월=12, 분기=4, 반기=2, 연=1 (연간 금액 = 해당기간 금액 × 승수)
- 수입 항목: 급여, 상여, 연금, 사업소득, 임대소득, 기타 — 각 항목에 압류/가압류 유무 표시
- 지출 항목: 생계비, 주거비, 의료비, 교육비, 기타 — 각 항목에 금액과 추가지출 사유 기재
- **추가생계비 비율 계산**: `((총지출 - 기본생계비) / 기본생계비) × 100` (반올림)
- 추가생계비 항목별 소명 요건: 각 항목에 영수증·계약서 등 소명자료를 첨부해야 한다

### 보증인·대위변제 처리 (REHABILITATION_LAW_RULES.md §7, 법원서식 D5130)

- **일부대위변제**: 보증인이 채무 일부를 변제한 경우
  - 원채권자 잔여채권 = 원금잔액 + 이자잔액 - 보증인변제액
  - 보증인 구상채권 = 보증인변제액 (일반 무담보 채권으로 별도 등재)
  - 채권자목록에 **2행** 기재해야 한다: (1) 원채권자 (감소된 금액), (2) 보증인 (구상채권)
- **전액대위변제**: 보증인이 원금 전액을 변제한 경우
  - 원채권자 잔여채권 = 이자만 남음 (원금 0)
  - 보증인 구상채권 = 원금 전액 (일반 무담보)
  - 채권자목록에 **2행** 기재해야 한다: (1) 원채권자 (이자만), (2) 보증인 (구상채권 = 원금)
- **장래구상권**: 보증인이 아직 변제하지 않은 경우의 장래 구상채권은 **미확정채권**으로 처리해야 한다
- **주채무-보증채무 간 변제 중복 방지**: 주채무자와 보증인의 채권을 합산할 때 동일 원금에 대해 이중 변제가 발생하지 않도록 해야 한다 — 보증채무의 안분 기준액은 주채무에서 이미 배분된 금액을 제외해야 한다

### 변제기간 6규칙 엔진 상세 (period-setting.ts)

`decidePeriodSetting()`은 6가지 설정에 따라 변제 대상과 기간을 결정한다:

```
설정1 (capital60):     원금만, 60개월, 이자 면책
설정2 (both60):        원금+이자, 60개월
설정3 (capital100_5y): 원금 100% + 잔여 이자, 최대 60개월
설정4 (capital100_3y): 원금 100% + 잔여 이자, 36개월
설정5 (both36):        원금+이자, 36개월
설정6 (capital36):     원금만, 36개월, 이자 면책 (기본값)
```

청산가치 보장 post-step:
- 기본 설정 후 `현재가치 < 청산가치`이면 변제기간을 48 → 60개월로 자동 연장해야 한다
- 60개월로도 부족하면 `reason='cap_60'` 경고를 반환해야 한다
- `forcedMonths` 파라미터로 사용자가 기간을 강제 지정할 수 있다

### 조세우선 계단식 배분 (schedule-generator.ts)

`tieredTaxPriority` 모드에서 조세 재단채권(has_priority_repay=true)을 단계적으로 배분해야 한다:

```
Phase 1 (조세 전액 구간):
  매월 변제액 전액을 조세 채권에 배분
  조세 채권 간에는 원금 비율로 안분
  기간: floor(조세총액 / 월변제액) 개월

Phase 2 (혼합 구간, 1개월):
  조세 잔여분 + 남은 금액으로 일반 안분 시작
  조세잔여 = 조세총액 - (Phase1 개월 × 월변제액)

Phase 3 (일반 단독 구간):
  매월 변제액 전액을 일반 채권에 pro-rata 안분
```

### capitalOnly 모드 (schedule-generator.ts)

`capitalOnly=true`이면 모든 스케줄 생성에서 이자를 무시해야 한다:

- 분모(totalDebt) = 원금 합계만 (이자 제외)
- 채권자별 분자(creditorDebt) = 원금만
- 배분 결과: `capitalRepay = total`, `interestRepay = 0` 고정
- `capitalOnly` 판별: `['capital36','capital60','capital100_3y','capital100_5y']` 옵션일 때 true
- 이 플래그는 `generateRepaySchedule`, `generateTieredSchedule`, `computePhasedRepaySegments` 모두에 전달해야 한다

### 월별 보정 스케줄 알고리즘 (rounding.ts)

`buildAdjustedSchedule()`로 월별 변제액의 올림 차이를 보정해야 한다:

```
basePayment = floor(monthlyAvailable)
totalTarget = 지정값 또는 basePayment × months
diff = totalTarget - (basePayment × months)

if diff > 0:
  adjustmentStartIndex = min(12, months - 1)
  adjustmentMonths = months - adjustmentStartIndex
  perAdjustment = ceil(diff / adjustmentMonths)

  1~12회차: basePayment (고정)
  13회차~마지막-1: basePayment + perAdjustment
  마지막 회차: 잔여분 흡수 (총합 = totalTarget 정확히 일치)
```

### 우선변제 강제 조정 메커니즘 (repayment-calculator.ts)

우선변제채권 100% 보장을 위해 월 변제액을 자동 상향 조정해야 한다:

```
if priorityDebt > 0:
  minMonthlyForPriority = ceil(priorityDebt / repayMonths)
  monthlyRepay = max(monthlyRepay, minMonthlyForPriority)

if monthlyRepay × repayMonths < priorityDebt:
  priorityInsufficient = true → 에러 표시
```

- 이 조정은 청산가치 보장 조정 **이전에** 적용해야 한다
- `RepaymentResult`에 `priorityDebt`, `priorityInsufficient` 필드로 결과를 반환해야 한다

### 채권자 분류 3경로 (creditor-classification.ts)

`classifyCreditor()`는 3가지 상호배타적 경로로 채권을 분류해야 한다:

```
경로 1: isOtherUnconfirmed = true
  → 전체 채권액(원금+이자) 미확정 처리, 확정액 = 0

경로 2: isSecured = true
  → 담보 평가액까지 확정, 초과분(deficiency) 미확정
  → deficiency = max(0, totalClaim - securedCollateralValue)
  → confirmedAmount = min(securedCollateralValue, totalClaim)

경로 3: 일반 (그 외)
  → 전체 채권액 확정, 미확정 = 0
```

- 경로 1과 경로 2는 상호배타이다 (isOtherUnconfirmed 우선)
- 이 분류 결과는 `generateTieredSchedule`에서 확정/미확정 배분 비율에 사용된다

### 기준중위소득 폴백 연도 (median-income.ts)

- 지원 연도: 2022, 2023, 2024, 2025, 2026
- 요청 연도가 지원 범위 밖이면 **2025년** 데이터로 폴백해야 한다

### 월별 스케줄 날짜 표시 (schedule-generator.ts)

- `startDate` 있음: `YYYY.MM` 형식 (월 2자리 0패딩)으로 각 회차 날짜를 계산해야 한다
- `startDate` 없음: `제N회` 형식으로 표시해야 한다
- 날짜 계산: `new Date(startDate).setMonth(month + (회차 - 1))`

### 별제권 순위 정렬 규칙 (secured-allocation.ts)

- 동일 담보물건에 복수 채권자가 있으면 `lienPriority` 오름차순으로 정렬해야 한다
- `lienPriority`가 미설정(null/0)이면 **999**로 간주하여 최하순위로 배치해야 한다
- 선순위 채권자부터 배분가능액을 소진하고, 후순위는 잔여분만 배분받는다

### 월가용소득 경고 임계값 (monthly-available.ts)

- `raw < 0`: "월가용소득이 음수입니다" 경고를 표시해야 한다
- `livingCostRate < 100`: "기준중위소득 60% 보장 위배 가능" 경고를 표시해야 한다
- `livingCostRate > 200`: "생계비율이 200% 초과 — 법원 이례적 승인 필요" 경고를 표시해야 한다

### 전자소송 CSV 양식 (법원 전자소송 시스템)

- 헤더: `채권자번호,채권자명,법인/개인,우편번호,도로명주소1,도로명주소2,전화번호,팩스번호,휴대전화번호,채권의원인,원금,이자`
- 인코딩: **UTF-8 BOM** (`\uFEFF` 선행) — Windows Excel 호환 필수
- 인격구분: `법인` / `개인` 이진 분류만 허용
- CSV 이스케이핑: RFC 4180 준수 (쉼표·따옴표·개행 포함 시 따옴표 감싸기)
