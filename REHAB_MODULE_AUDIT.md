## 1. 채권자 가지번호 미적용 (처리지침 §5)

**심각도**: 🔴 법원 서식 위반

- C저축은행(4번, 보증인) → 4-1 없음
- F저축은행(6번, 보증인) + 이대위(7번, 대위변제) → 6과 6-1이어야 함
- G캐피탈(8번, 보증인) + 박전액(9번, 대위변제) → 8과 8-1이어야 함

- `formatBondNumber()` (rehab-creditors-tab.tsx:67-73) — 출력 로직은 정상
- **문제**: UI에서 `parent_creditor_id`를 연결하는 필드가 없거나, 더미 데이터에 parent 관계가 설정 안 됨

### CLAUDE.md 규칙 (line 342-347)
```
- 보증채무/연대보증 → 주채무자 채권번호의 하위 번호 (예: 1-1, 1-2, 1-3)
```

## 2. 저장 버튼 패턴 위반 (전 탭)

**심각도**: 🟡 UX 규칙 위반

### 현상
5개 탭 모두 `ClientActionForm` + `SubmitButton` 대신 커스텀 `<button onClick={handleSave}>` 사용.

### 위반 파일 목록
| 탭 | 파일 | 위반 라인 |
|---|---|---|
| 신청인 | rehab-applicant-tab.tsx | 1095-1104 |
| 채권자 | rehab-creditors-tab.tsx | 1148-1157 |
| 재산 | rehab-property-tab.tsx | 368-377 |
| 소득/생계비 | rehab-income-tab.tsx | 745-754 |
| 진술서 | rehab-affidavit-tab.tsx | 507-518 |

### CLAUDE.md 규칙 (line 89-96)
```tsx
// ❌ 금지
<form action={serverAction}><button type="submit">저장</button></form>

// ✅ 필수
<ClientActionForm action={serverAction} successTitle="저장 완료">
  <SubmitButton>저장</SubmitButton>
</ClientActionForm>
```

### 현재 구현 패턴 (5개 탭 동일)
```tsx
// ❌ 실제 코드 — 수동 상태 관리 + 수동 토스트
<button type="button" onClick={handleSave} disabled={saving}>
  {saving ? '저장 중...' : '저장'}
</button>
```

### 참고
- 토스트 자체는 `useToast()`로 수동 호출하고 있어 기능적으로는 동작하지만, `ClientActionForm` 자동 연결 패턴을 따르지 않음
- 브라우저 테스트 시 저장 클릭 후 토스트가 보이지 않았던 건 이 수동 패턴 문제일 가능성 있음

### 수정 방향
- 개인회생 탭은 멀티 upsert(여러 테이블 동시 저장)라 단순 `ClientActionForm` 1:1 매핑이 어려울 수 있음
- 최소한 **토스트 피드백이 확실히 동작하는지** 검증 필요
- 가능하면 `ClientActionForm` 패턴으로 전환, 불가하면 수동 패턴의 토스트 동작 보장

## 9. 기준중위소득 60% — 공표값 미사용 (Math.floor 계산)

**심각도**: 🟡 데이터 정확성

### 현상
- 기준중위소득 60%는 보건복지부가 **공표하는 고정 숫자**
- 현재 코드: `Math.floor((기준중위소득100% × 60) / 100)` → 소수점 버림
- 결과: 공표값보다 **1원 낮은 값** 산출 (거의 모든 가구원 수에서)
- 예: 2026년 1인 → 2,564,238 × 60% = 1,538,542.8 → `floor` = 1,538,542원
- **채무자에게 불리**: 생계비 1원↓ → 가용소득 1원↑ → 변제액 1원↑

### 수정 방향
- 계산하지 말고 **공표된 정확한 값을 테이블에 하드코딩**
- `median-income.ts`에 60% 값 테이블을 별도로 두거나, 공표값을 직접 입력
- 법원 전산(COLAW)과 수치 일치가 목표

**심각도**: 🟡 UX 설계 오류

### 현상
- **월 소득**: 수동 직접 입력 필드 (form.monthly_income)
- **수입 명목별 상세 (D5103)**: 급여/상여/연금 등 항목별 입력 → 연간환산 → 월 평균 수입 자동 계산
- **두 섹션이 완전히 분리** — D5103 합계가 월 소득 필드에 반영 안 됨
- 사용자 혼란: 어느 값이 실제 변제계획에 쓰이는지 불명확

### CLAUDE.md 기준
- D5103(수입 및 지출에 관한 목록)은 법원 제출 서식
- 월 소득은 변제계획 계산의 입력값
- **둘 중 하나만 있어야 하거나, D5103 합계 → 월 소득 자동 반영이 맞음**

### 수정 방향 (택1)
- **A안**: D5103 명목별 합계를 월 소득에 자동 반영 (D5103이 원본, 월 소득은 읽기전용)
- **B안**: 월 소득 직접 입력만 남기고, D5103은 문서 출력 시에만 참조
- **권장**: A안 — D5103이 법원 제출 서식이므로 이를 원본으로 삼는 게 정합성에 맞음
